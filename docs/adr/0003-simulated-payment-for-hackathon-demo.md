# מסך תשלום (מסלול רכישת תמהיל) מדומה לדמו, לא אינטגרציית סליקה אמיתית

**Status:** accepted

מסלול "רכישת תמהיל + הוצאת אישורים" כולל לפי המסמך "מסך תשלום" לפני המשך התהליך. לצורך ההאקתון, נבנה את המסך/הזרימה במלואה (UI מלא, Cloud Function שמסמן `paymentStatus: 'simulated_paid'` על הבקשה ב-Firestore), **בלי** אינטגרציה אמיתית לספק סליקה (Tranzila/Cardcom/PayPlus/Stripe).

**הנימוק:** הקמת חשבון סליקה אמיתי (כולל אישורי PCI/sandbox) לא משתלמת בחלון הזמן של ההאקתון, והמחוון מדגיש מיקוד בעיקר (25% "הצגת העיקר מתוך הדמו") ולא בפרטי תשתית תשלומים. ההחלטה היא מכוונת ולא פער שנשמט — חשוב לתעד את זה כדי שמעריך שיבדוק את הקוד לא יחשוב שזה נשכח.

**להמשך (אחרי ההאקתון):** להחליף את ה-Cloud Function המדומה באינטגרציה אמיתית (PayPlus מומלץ לשוק הישראלי — hosted page + webhook, פחות מורכב מ-Tranzila/Cardcom הקלאסיים) בלי לשנות את חוזה ה-API שצרכני המסך כבר תלויים בו.
